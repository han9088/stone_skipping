# 📄 PRD：打水漂体感模拟器（Stone Skip Simulator）— 无引擎 · 伪 2D 版

> 产品需求与 [PRD.md](./PRD.md) 一致，本文档为 **不使用游戏引擎**、**无 3D 回放**、采用 **伪 2D 视觉** 的技术实现版本。

---

## 1. 产品概述

### 1.1 产品名称

Stone Skip / 打水漂模拟器（暂定）

### 1.2 产品定位

基于手机体感（加速度 + 陀螺仪）的投掷模拟应用。  
主玩法靠 **震动节奏** 感知水漂次数，无需盯屏；可选观看 **伪 2D 侧视回放** 重温轨迹。

用户通过「模拟打水漂动作」完成一次投掷，系统在 2D 画布上回放石头掠过水面的动画（非真实 3D）。

### 1.3 支持平台

| 平台 | 最低版本 | 说明 |
|------|----------|------|
| **iOS** | iOS 14+ | iPhone（需加速度计 + 陀螺仪） |
| **Android** | Android 8.0 (API 26)+ | 主流机型（需加速度计 + 陀螺仪） |

- V1 同步支持 **Android** 与 **iOS** 双端发布
- 采用 **跨平台 App 框架**（推荐 Flutter 或 React Native / Expo），**不使用 Unity / Godot 等游戏引擎**
- 平台差异体现在传感器插件、震动 API、权限与商店发布流程

### 1.4 核心体验

用户无需持续观看屏幕，核心分 **两个阶段**：

**阶段一 · 投掷与体感（主玩法）**

甩动手机 → 识别「出手」→ 实时模拟水漂 → 每个落点震一次 → 水漂结束展示结果

**阶段二 · 伪 2D 回放（可选）**

水漂全部结束后，用户主动进入 **侧视 2D 动画回放**，重温轨迹与水面特效（精灵 / 矢量 / 粒子，非 3D 场景）。

> 震动发生在 **扔出之后的实时模拟阶段**，不在 2D 回放中重复震动。

### 1.5 与引擎版 PRD 的差异

| 维度 | 引擎版（Unity / Godot） | 本版（无引擎 · 伪 2D） |
|------|-------------------------|--------------------------|
| 运行时 | 游戏引擎 + 3D 场景图 | 普通 App + 2D 绘制层 |
| 回放 | 3D 镜头、Mesh、粒子 | 侧视 2D 精灵动画 + 视差 |
| 包体 | 通常 30–80 MB+ | 目标 < 15 MB |
| 开发重心 | 场景 / 材质 / 相机 | 传感器 + 物理 + Canvas |
| 主反馈 | 震动（相同） | 震动（相同） |

## 2. 核心玩法

### 2.1 用户流程

#### 阶段一：投掷 + 实时反馈（无需看屏幕）

1. 手持手机（模拟石头，**不可真的扔出手机**）
2. 做打水漂甩动动作
3. 系统识别「出手瞬间」，提取投掷参数
4. **实时**运行物理模拟，按落点时间线依次震动
5. 水漂结束（石头下沉），展示结果（次数 / 评分）

#### 阶段二：伪 2D 回放（可选，水漂结束后）

6. 用户点击「查看回放」
7. 播放 **侧视 2D** 打水漂动画（石头精灵沿轨迹运动、水面波纹、水花帧动画）
8. 回放纯视觉，**不再触发震动**

> 实时模拟阶段可显示极简 UI（次数计数、进度条），也可黑屏 / 待机页，与引擎版一致。

### 2.2 体验时序

```
[待机] → [甩动] → [识别出手] → [实时模拟 + 落点震动 × N] → [结果页] → [可选：伪 2D 回放]
         ↑ 用户操作      ↑ 难点              ↑ 主反馈                      ↑ 2D 侧视动画
```

### 2.3 伪 2D 视觉定义

「伪 2D」指 **逻辑与绘制均在 2D 平面**，通过分层与动效营造纵深，**不使用 3D 引擎、Mesh、Camera3D**：

| 元素 | 表现方式 |
|------|----------|
| 天空 / 远山 | 背景图层，慢速视差滚动 |
| 水面 | 水平色带 + 正弦波顶点动画（Shader 或 Canvas 路径） |
| 石头 | 单张精灵或简单椭圆，沿 `(x, y)` 轨迹运动 |
| 水花 | 2D 帧动画序列 或 径向粒子（Canvas / Skia） |
| 波纹 | 同心圆缩放 + 透明度淡出 |
| 镜头 | 2D 相机：横向跟随石头 `scrollX`，无 3D 旋转 |

视角固定为 **侧视（Side View）**，与真实打水漂观察角度一致，实现成本低于 3D 跟随镜头。

## 3. 核心功能模块

### 3.1 动作采集模块（Motion Capture）

#### 输入设备

- 加速度计（Accelerometer）
- 陀螺仪（Gyroscope）

#### 功能

- 高频采样（目标 60Hz+，尽力接近 100Hz，受 OS 与插件限制）
- 记录最近 1~2 秒动作数据（环形缓冲区）
- 捕捉运动峰值
- 跨平台统一数据格式

#### 技术实现（框架无关）

| 框架 | 推荐插件 / API |
|------|----------------|
| **Flutter** | `sensors_plus`（加速度、陀螺仪、用户加速度） |
| **React Native / Expo** | `expo-sensors`（Accelerometer、Gyroscope） |
| **原生** | iOS CoreMotion / Android SensorManager（双端各写一层，上层 TypeScript / Dart 抽象） |

#### 输出数据

| 参数 | 说明 |
|------|------|
| velocity | 速度 |
| angle | 出手角度 |
| spin | 旋转 |
| sideSpin | 横向旋转 |

> 坐标系需在 Dart / TS 层归一化（设备坐标 → 投掷坐标），与引擎版逻辑相同，与是否使用游戏引擎无关。

### 3.2 动作识别模块（Throw Detection）

#### 核心难点

**如何判断「石头已扔出去」，但用户并没有真的把手机扔出去？**

与 [PRD.md](./PRD.md) 完全一致：从握持状态下的 **运动曲线** 推断虚拟出手时刻，不依赖脱手检测。

#### 设计原则

- 识别的是 **「模拟出手」**，不是检测手机脱手
- 用户全程握持手机，通过甩动轨迹的峰值与突变来锁定参数
- 误识别需有兜底：无效动作提示重试，避免误触发模拟

#### 判定逻辑（V1 方案）

| 阶段 | 传感器特征 | 动作 |
|------|------------|------|
| 待机 | 加速度接近重力静止 | 等待甩动 |
| 甩动开始 | 加速度超过启动阈值 | 开始记录 1~2s 滑动窗口 |
| 虚拟出手 | 加速度达峰值后 **急剧回落** | 锁定峰值时刻的参数作为「出手」 |
| 确认 | 角速度方向 + 水平分量符合投掷姿态 | 冻结数据，进入模拟 |
| 无效 | 未达阈值 / 窗口超时 / 姿态不符 | 提示「再试一次」，不进入模拟 |

#### 出手参数提取

- **速度**：出手前短窗口内加速度积分峰值
- **角度**：出手时刻设备 pitch（入水角）
- **旋转**：出手时刻 gyro 角速度（spin / sideSpin）

#### 校准与容错

- 阈值写入配置文件（JSON / 远程配置），便于真机热更新
- 支持「练习模式」降低识别门槛，正式模式提高精度

#### 实现建议

- **纯 Dart / TypeScript 实现**，与 UI 框架解耦，便于单元测试
- 桌面调试时用录制的传感器 CSV 回放，无需真机甩动

---

### 3.3 简化物理模拟（Stone Physics Engine）

> 不使用真实流体模拟；纯参数模型，与引擎版公式一致

#### 运行模式

| 模式 | 时机 | 输出 |
|------|------|------|
| **实时模拟** | 识别出手后立即运行 | 落点时间线 → 驱动震动；完成后输出 Skips / Score |
| **2D 回放渲染** | 水漂结束后用户主动进入 | 读取已生成 Trajectory，驱动 2D 动画 |

#### 核心逻辑

- 入水角 < 20° 且速度足够 → 触发水漂
- 每次水漂 → 速度衰减
- 超过阈值 → 下沉

#### 输出

- 水漂次数（Skips）
- 轨迹点 `TrajectoryPoint[]`：`{ t, x, y, event: 'skip' | 'splash' | 'sink' }`
- 2D 坐标系：x 为水平距离，y 为离地高度（侧视）

### 3.4 伪 2D 回放系统（2D Replay）

> **定位**：水漂结束后的可选视觉重温，非主玩法必经步骤。  
> **替代** 引擎版 3D 回放，无 3D 场景、无 Cinemachine、无 Mesh。

#### 触发条件

- 实时模拟完成、结果页展示后
- 用户点击「查看回放」进入
- 回放期间 **不触发震动**

#### 技术方案

| 框架 | 2D 绘制方案 |
|------|-------------|
| **Flutter** | `CustomPainter` + `AnimationController`；或 `flame`（仅作 2D 游戏循环，非完整 3D 引擎） |
| **React Native** | `react-native-skia` 或 `react-native-reanimated` + SVG |
| **Expo** | `expo-gl` + 2D 正交投影（不推荐，优先 Skia / Canvas） |

> **推荐**：Flutter `CustomPainter` 或 Flame 2D 模式 — API 简单、性能足够、不引入 3D 管线。

#### 回放流程

1. 加载 `Trajectory` 与落点事件列表
2. 初始化侧视场景：天空层、水面线、石头精灵
3. 按时间轴 `t` 插值石头位置 `(x, y)`
4. 触水事件 → 播放水花帧动画 + 波纹 Tween
5. 相机 `scrollX` 跟随石头（带平滑阻尼）
6. 结束后展示「再玩一次」/ 返回结果页

#### 镜头与效果（均为 2D）

| 效果 | 实现 |
|------|------|
| Follow | `scrollX = lerp(scrollX, stone.x - viewport/2)` |
| 慢动作 | 回放时间轴 `playbackSpeed = 0.5` |
| 水花 | 精灵表 4–8 帧 或 10–20 个 2D 粒子 |
| 波纹 | `scale` 1→3 + `opacity` 1→0，约 400ms |
| 视差 | 背景层 `scrollX * 0.3`，近景 `scrollX * 0.8` |

#### 性能目标

- 60 FPS（主流机型）
- 同屏粒子 < 50（2D）
- 无 3D draw call、无骨骼动画

### 3.5 水面与特效系统（Water FX · 2D）

#### 特效（全部为 2D）

| 特效 | 说明 |
|------|------|
| Splash | 触水瞬间精灵帧动画或径向粒子 |
| Ripple | 水面同心圆，scale + fade |
| Foam | 可选，短生命周期小圆点 |

#### 触发方式

| 模式 | 画面 | 特效 |
|------|------|------|
| **实时模拟** | 极简 UI 或无画面 | 无（仅震动） |
| **2D 回放** | 完整侧视场景 | 每次触水播放 Splash + Ripple |

#### 美术风格建议

- 扁平 / 插画风侧视湖面，降低资产成本
- 石头：单 PNG 或程序绘制圆角矩形
- 避免依赖 3D 模型与 PBR 材质

### 3.6 震动反馈系统（Haptics）

> 与 [PRD.md](./PRD.md) 完全一致：震动是 **主玩法反馈**，发生在实时模拟阶段，2D 回放不重复震动。

#### 核心规则

**一个水漂落点 → 手机震动一次（实时，非回放）**

#### 触发点

| 时机 | 震动次数 | 说明 |
|------|----------|------|
| 虚拟出手确认 | 1 次 | 识别出手成功、开始实时模拟时 |
| 每个水漂落点 | 各 1 次 | 实时模拟中触水时刻 |
| 2D 回放 | 0 次 | 回放仅视觉 |

#### 平台实现

| 框架 | API |
|------|-----|
| **Flutter** | `vibration` / `haptic_feedback` |
| **Expo** | `expo-haptics`（`impactAsync` / `notificationAsync`） |
| **iOS 原生** | Core Haptics / `UIImpactFeedbackGenerator` |
| **Android 原生** | `VibrationEffect.createOneShot` |

> V1.1 可选：自定义震动波形；V1 保持「一落点一短震」即可。

### 3.8 UI 层（App Shell）

无引擎版以 **标准 App 导航** 组织页面，非 Scene 切换：

| 页面 | 内容 |
|------|------|

## 4. 技术架构

```
┌─────────────────────────────────────────┐
│     跨平台 App（Flutter / Expo）          │
├─────────────────────────────────────────┤
│  Platform Plugins（传感器 / 震动）        │
└─────────────────┬───────────────────────┘
                  ↓
      MotionCapture（高频采样 + 环形缓冲）
                  ↓
    ThrowDetector（虚拟出手识别 ★难点）
                  ↓
      StonePhysicsEngine（纯 Dart/TS）
                  ↓
    HapticsScheduler（落点 → 震动，主反馈）
                  ↓
      TrajectoryStore（JSON / 本地 DB）
                  ↓
           UI Shell（路由 + 结果页）
                  ↓
    Replay2DView（CustomPainter / Skia，纯视觉）
```

**核心原则**：业务逻辑（识别 + 物理 + 震动）与渲染层完全分离，2D 回放仅消费 `Trajectory` 数据。

## 5. 技术选型

### MVP 推荐（二选一）

#### 方案 A：Flutter（推荐）

| 类别 | 选型 |
|------|------|
| 框架 | **Flutter 3.x** |
| 语言 | **Dart** |
| 传感器 | `sensors_plus` |
| 震动 | `vibration` 或 `haptic_feedback` |
| 2D 回放 | `CustomPainter` + `AnimationController` |
| 状态 | `riverpod` 或 `bloc` |
| 存储 | `shared_preferences` + JSON |
| 构建 | `flutter build ipa` / `flutter build appbundle` |

**优点**：单语言、绘制 API 成熟、包体小、无 JS 桥接延迟。  
**缺点**：团队需熟悉 Dart。

#### 方案 B：React Native + Expo

| 类别 | 选型 |
|------|------|
| 框架 | **Expo SDK 52+** |
| 语言 | **TypeScript** |
| 传感器 | `expo-sensors` |
| 震动 | `expo-haptics` |
| 2D 回放 | `react-native-skia` + Reanimated |
| 状态 | `zustand` |
| 构建 | EAS Build → TestFlight / Play |

**优点**：Web 技术栈、热更新友好、Maestro 测试成熟。  
**缺点**：Skia 包体积与原生桥接需额外调优。

### 明确不采用

| 技术 | 原因 |
|------|------|
| Unity / Godot / Unreal | 用户要求无游戏引擎 |
| 3D 回放（任意形式） | 用户要求无 3D 回放 |
| Web-only PWA | 传感器与后台震动能力弱，不适合主玩法 |

### 平台发布

| 平台 | 构建目标 | 分发渠道 |
|------|----------|----------|
| Android | AAB / APK | Google Play / 侧载 |
| iOS | IPA / Xcode | TestFlight / App Store |

### 平台权限

| 平台 | 配置 | 用途 |
|------|------|------|
| Android | `VIBRATE` | 落点震动 |
| iOS | 无额外权限（传感器默认） | 加速度计 / 陀螺仪 |
| 双端 | 可选运动与健身说明（App Store 审核文案） | 传感器用途披露 |

### 与引擎版对比

| 项 | 无引擎 · 伪 2D | Unity / Godot |
|----|----------------|---------------|
| 包体 | ~10–20 MB | ~40–80 MB+ |
| 回放视觉 | 侧视 2D 精灵 | 3D 场景 + 粒子 |
| 主玩法 | 相同（震动） | 相同（震动） |
| 学习曲线 | App 开发 + Canvas | 引擎 + 3D 工作流 |
| 迭代速度 | 逻辑改完即测，无 Scene 合并 | 场景与脚本协同 |

## 10. 风险

### 技术风险

| 风险 | 说明 |
|------|------|
| 虚拟出手识别 | **核心难点**：与框架无关，需真机调参 |
| 传感器采样率 | Flutter / RN 插件频率低于原生，需实测并做插值 |
| 后台 / 锁屏 | 甩动时屏幕可能熄灭，震动调度须不依赖 UI 线程 |
| 2D 回放观感 | 无 3D 纵深感，需靠动效与节奏补偿 |
| JS 桥接（若用 RN） | 高频传感器需评估是否写原生 Module |

### 产品风险

| 风险 | 说明 |
|------|------|
| 玩法疲劳 | 单一核心玩法可能导致留存下降 |
| 真实 vs 趣味 | 物理精度与游戏性平衡 |
| 回放价值 | 用户可能只看震动结果、跳过 2D 回放 |

### 缓解策略

- 主玩法不依赖画面，回放为锦上添花
- 物理与识别模块独立测试，不绑 UI
- 优先 Flutter 方案降低传感器链路复杂度

---

## 11. 一句话总结

用手机模拟投掷，识别虚拟出手后实时震出水漂节奏；水漂结束后，可选观看 **伪 2D 侧视回放** — 全程 **无游戏引擎、无 3D**。
